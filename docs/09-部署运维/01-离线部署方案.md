# 01 离线部署方案

## 1. 总体策略

- **生产离线**：学校服务器部署时，所有安装介质 / 镜像 / 字典 / 字体 / GeoJSON / 依赖包 均本地化；开发测试阶段允许联网构建与联调；
- **国产化兼容**：麒麟 V10 SP3、统信 UOS V20（x86_64 首要，ARM 兼容预留）；
- **单校单实例**：无多租户；生产环境为必选，预发/开发环境按学校资源条件选配；
- **部署形态**：Docker 24 + Docker Compose。

## 3. 部署前提条件

### 3.1 硬件
参见 `01-总体架构/04-部署架构与网络设计.md#8`。

### 3.2 网络
- 所有节点时间同步（NTP 内网源）；
- 按 VLAN 划分好网段；
- 节点间内网可达，外网完全不需要；
- 负载均衡器（F5 / HAProxy / LVS + Keepalived）准备 VIP。

### 3.3 系统初始化（每节点）
- 保留 `firewalld`，按白名单开放端口；
- SELinux 安装阶段可临时置为 `permissive` 便于排障，验收前需恢复为 `enforcing` 并固化策略；
- 关闭 swap；
- 内核参数：
  ```
  vm.max_map_count=262144
  fs.file-max=2097152
  net.core.somaxconn=65535
  net.ipv4.ip_local_port_range=1024 65535
  net.ipv4.tcp_tw_reuse=1
  ```
- ulimit：nofile=1048576、nproc=65535；
- 安装基础包：`gcc, make, tar, unzip, jq, rsync, chrony, socat, conntrack`。

### 3.4 账号
- 部署账号 `bdc-deploy`（sudo 权限）；
- 应用账号 `bdc-runner`（nonroot）。

## 4. 部署步骤（Docker Compose）

### 4.1 环境初始化
```bash
bash scripts/00-check-env.sh
bash scripts/10-install-os-deps.sh        # 各节点并行
```

### 4.2 容器运行时
```bash
bash scripts/20-install-container.sh       # Docker 24 + containerd
```

### 4.3 Harbor 私仓
```bash
bash scripts/30-install-harbor.sh --host harbor.bdc.edu.cn
bash scripts/40-import-images.sh           # 导入所有镜像 tar
```

### 4.4 基础中间件
```bash
bash scripts/60-install-infra.sh \
    --pg 3 --ck 3 --kafka 3 --minio 4 \
    --redis 3 --elasticsearch 3
```
- 以 Docker Compose + 独立 compose 文件安装；
- 数据目录挂载到物理 NVMe/SSD；
- 持久化：使用宿主机目录或独立数据盘挂载。
- Flink 一期默认采用 checkpoint/savepoint + 调度重拉起，不额外引入 ZooKeeper。
- Flowable 以 `bdc-approval` 服务内嵌方式交付，无需单独部署镜像。

### 4.5 监控栈
```bash
docker compose -f compose/deps/observability.yml up -d
# 内含 prometheus + grafana + loki + skywalking
```

### 4.6 数据库初始化

> 一期物理库口径（与 `02-数据架构/02-数据库分库与表结构规范.md`、`04-数据库详细设计/01-核心表结构清单.md` 保持一致）：
> - PG 业务实例：`bdc_app`（按 Schema 隔离 18 个业务服务）、`bdc_flowable`（审批，Flowable 官方表 + 业务映射表）、`bdc_audit`（审计，独立库 + 只追加表）；
> - PG 数仓实例：`bdc_dw`（含 `ods_*` / `dwd_*` / `dim` 等 Schema）；
> - ClickHouse：`bdc_dw_olap`（含 DWS / ADS 表）。
>
> 业务服务的 Schema 名直接复用服务名，例如 `bdc-uac` → `bdc_app.uac`、`bdc-visual` → `bdc_app.visual`，文档中出现的 `bdc_uac.t_user`、`bdc_visual.t_page` 是“逻辑模块 Schema 名”的简写。

```bash
# 1) 创建 5 个物理库（bdc_app / bdc_flowable / bdc_audit / bdc_dw + 各服务专用账号）
psql -h pg-master -U postgres -f sql/00-create-dbs.sql

# 2) 初始化 bdc_app 库内 18 个业务 Schema
for svc in auth uac metadata mdm indicator tag collect warehouse govern push \
           dataservice market visual security notify config; do
  psql -h pg-master -U bdc_app_owner -d bdc_app -f sql/01-app/$svc/
done

# 3) 初始化独立库
psql -h pg-master -U bdc_flowable_owner -d bdc_flowable -f sql/01-flowable/  # Flowable 官方建表 + bdc_flowable.t_biz_flow_map
psql -h pg-master -U bdc_audit_owner    -d bdc_audit    -f sql/01-audit/     # 审计只追加表 + 触发器

# 4) 数仓 PG（独立实例，与业务 PG 物理隔离）
psql -h pg-dw -U bdc_dw_owner -d bdc_dw -f sql/02-dw/

# 5) ClickHouse 数仓
clickhouse-client --host ck-master --multiquery < sql/03-ck/init.sql
```

### 4.7 中台微服务
```bash
docker compose -f compose/platform/docker-compose.prod.yml up -d
```
- 按依赖顺序：`uac/auth/metadata` 先行；
- Nacos 首次启动后通过导入脚本加载配置。

### 4.8 Nacos 配置导入
```bash
python3 scripts/nacos-import.py --namespace prod --from config/env/prod/
```

### 4.9 验证
```bash
bash scripts/80-post-check.sh
```
- 健康检查 20 个服务 `/actuator/health`；
- 可执行最小业务流程：CAS 登录、资产检索、一次 API 调用。

## 5. 配置管理

- 所有配置在 Nacos 的命名空间中维护；
- 敏感配置使用 `ENC(...)` 包装，启动时 KMS 解密；
- 按环境三套命名空间：`prod/test/dev`。

## 6. 升级策略

- **蓝绿**：两套 Compose 服务栈并行启动 + Nginx/Gateway 切换；
- **金丝雀**：基于 Header / Weight 10%→30%→100%；
- **回滚**：回切上一版镜像 tag 并重新 `docker compose up -d`；
- **数据库变更**：Flyway 在新版本启动时执行（阻止错误版本启动）。

## 7. 许可证与国密合规

- 软件许可：开源项目一并附带 LICENSE；
- 国密：使用通过国密局认证的实现（如 Tongsuo/BabaSSL/Bouncy Castle + 扩展包）；
- 审计合规：所有变更进入审计日志；部署动作经堡垒机留痕。

## 8. 常见问题排查

| 现象 | 排查点 |
| --- | --- |
| 服务启动失败 | 查 `docker logs`、`/actuator/health` 详情、Nacos 配置 |
| Kafka 写入慢 | 副本数、IO 是否瓶颈、broker CPU |
| CK 查询慢 | 分区/排序键、skip index、TTL |
| PG 锁等待 | `pg_stat_activity`、`pg_blocking_pids` |
| MinIO 403 | 桶策略、Key 是否过期 |
| 跨 VLAN 不通 | 防火墙策略、路由 |
| SSO 死循环 | CAS Service 注册、回调地址一致 |
