# 01 数据分层与分域

## 1. 分层整体架构

```
┌────────────────────────────────────────────────────────────────────┐
│                      数据源（业务系统 / 外部 API / 文件 / MQ）      │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ Flink CDC / DataX / API 采集
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ ODS 贴源层  (PostgreSQL) - 保留源结构, 默认不分区/大表按月分区, 软删标识 │
│   特点: 字段名/类型保持源端一致; 仅加系统字段(批次/时间/来源)         │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ 质量清洗 / 标准化 / 脱敏 / 口径统一
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ DWD 明细层  (PostgreSQL) - 按业务过程建事实表, 字段规范化             │
│                                                                     │
│ DIM 维度层  (PostgreSQL) - 一致性维度 (师/生/部门/时间/课程/专业…)    │
│   SCD2 拉链 + 版本管理; 维度代码表 / 缓慢变化历史                     │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ 聚合 / 宽表 / 指标预计算
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ DWS 汇总层  (ClickHouse) - 按主题轻度/中度聚合, 宽表模型              │
│   如: 学生画像宽表、教学运行汇总、财务收支汇总                         │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ 场景化加工
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ ADS 应用层  (ClickHouse) - 直供大屏/BI/API 的成品表                  │
│   如: 校领导驾驶舱、专业招生看板、疫情日报(示例)                      │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│ 非结构化数据湖 (MinIO + PostgreSQL 元数据索引)                       │
│   文件类型: 图片、音视频、档案、扫描件                                │
│   元数据: 哈希 SM3/SHA256、分级、水印、标签、关联业务主键             │
└────────────────────────────────────────────────────────────────────┘
```

## 2. 分层规范

### 2.1 ODS（Operational Data Store）

- **目的**：贴源、可追溯。
- **存储位置**：PostgreSQL，Schema 命名 `ods_<源系统>`，如 `ods_jwxt`（教务）、`ods_xgxt`（学工）。
- **表命名**：`ods_<源系统>_<源表>`；如 `ods_jwxt_t_student`。
- **分区规则**：ODS 默认**不分区**，仅当单表数据量持续增长到管理和查询成本明显升高时，再按 `_etl_ts` 做**月 RANGE 分区**；一期不采用“按天建分区表”方案。
- **字段规范**：
  - 保持源字段名、类型（中文列名一律改英文，映射入 `bdc_metadata.col_map`）；
  - 必加系统字段： 
    - `_src_id`（源记录主键）
    - `_batch_id`（采集批次 UUID）
    - `_op` (I/U/D，CDC 操作类型)
    - `_etl_ts`（入库时间 timestamptz）
    - `_src_ts`（源端变更时间，若可获得）
    - `_is_deleted`（逻辑删除，默认 false）
- **写入策略**：CDC 实时；拉链同步（SCD2）；全量覆盖。
- **保留策略**：默认 **2 年**；支持按表覆盖。

### 2.2 DWD（明细层）

- **目的**：清洗规范化、口径统一、质量达标；业务过程粒度的事实表。
- **Schema**：`dwd_<主题域>`，如 `dwd_teach`、`dwd_stu`、`dwd_fin`。
- **表命名**：`dwd_<主题>_<业务过程>_<周期>`；如 `dwd_teach_course_select_d`（选课，日粒度）。
- **建模**：维度建模；事实表关联 DIM 维度主键（代理键）。
- **清洗**：空值、脏值、编码统一（性别 F/M、民族、国家、行政区划）、时间规范化（UTC+8）、手机号格式化、去重、合并。
- **脱敏**：按字段分级表（`bdc_security.t_col_class`）在 DWD 写入阶段执行（详见 `06-安全与合规/03-脱敏与加密.md`）。

### 2.3 DIM（维度层） 

- **目的**：全局一致性维度。
- **Schema**：`dim`。
- **一致性维度**：
  - `dim_person`（师生全员）：`person_sk`（代理主键）、`person_id`（身份证/工号/学号 Hash）、`person_type`、`name`、`gender`、`nation`、`birth_date`、`dept_sk`、`effect_from`/`effect_to`、`is_current`
  - `dim_dept`（部门/学院/专业）：树结构 + 代理主键
  - `dim_course`（课程）
  - `dim_date`（日期维度）
  - `dim_term`（学期、学年、校历）
  - `dim_building`（楼栋教室）
  - ...
- **SCD 策略**：SCD2 为默认；当属性不敏感时可用 SCD1；大文本字段用 SCD3。

### 2.4 DWS（汇总层）

- **存储**：ClickHouse，表引擎 `ReplicatedReplacingMergeTree` / `AggregatingMergeTree`。
- **目的**：主题宽表 / 预聚合。
- **命名**：`dws_<主题>_<对象>_<粒度>_<周期>`，如 `dws_stu_profile_w`（学生画像宽表，周粒度）、`dws_teach_course_m`（教学课程汇总，月粒度）。

### 2.5 ADS（应用层）

- **存储**：ClickHouse；
- **命名**：`ads_<业务场景>`，如 `ads_leader_dashboard_kpi_d`；
- **特点**：直供大屏/报表；少关联、查询简单。

### 2.6 非结构化数据湖

- 物理存储：MinIO，桶命名 `bdc-<业务>-<分级>`，如 `bdc-archive-l3`；
- 元数据表：`bdc_warehouse.t_file_asset`，关联业务主键与分级；
- 查询检索：Elasticsearch 建文件元数据索引（含 OCR 文本提取可选）。

## 4. 主数据 / 维度数据对应

- 主数据（MDM） = 全校唯一权威源，用于 DIM 的"源数据"；
- MDM 生成的唯一 ID（ORG\_ID / STAFF\_ID / STUDENT\_ID / COURSE\_ID）写入 DIM 表的自然键；
- MDM 更新事件 `mdm.sync` → 各 ODS/DWD 表加工时校正。

## 5. 命名与 Schema 规约

| 位置         | 约定                                                                                 |
| ---------- | ---------------------------------------------------------------------------------- |
| Schema（PG） | `ods_*` / `dwd_<domain>` / `dim` / `ads_*`（中台中 ADS 放在 CK） / `bdc_<service>`（业务服务库） |
| 表名         | 小写 + 下划线；尽量完整表达业务含义                                                                |
| 字段名        | 小写 + 下划线；英文为主；避免缩写歧义                                                               |
| 布尔字段       | `is_xxx` / `has_xxx`                                                               |
| 时间字段       | `xxx_time`（timestamp）/`xxx_date`（date）                                             |
| 主键         | `<表名单数>_id` 或 `id`（BIGINT / UUID）                                                  |
| 代理主键       | `_sk`（Surrogate Key）                                                               |
| 外键引用       | `<外表单数>_id`                                                                        |

## 6. 数据流动总览（简化）

```
业务库                  ODS                     DWD/DIM                DWS/ADS
(PG/MySQL/Oracle ..)    (PostgreSQL)            (PostgreSQL)          (ClickHouse)

Flink CDC  ────▶ ods_<src>_<table>  ─SQL批─▶ dwd_<domain>_<biz>  ─SQL/Flink─▶ dws_*, ads_*
DataX 全量 ────▶           ↑                    dim_*                              │
API/文件   ────▶   (根据源选择)                  ↑                                  │
                                              MDM 表 ─SCD2─┘                       ▼
                                                                          大屏/BI/API
```

## 7. 数据生命周期

| 层       | 热数据           | 温数据                   | 冷数据                          | 归档           |
| ------- | ------------- | --------------------- | ---------------------------- | ------------ |
| ODS     | ≤ 90 天（PG 主库） | 90d\~1y（PG 普通表或月分区归档） | 1\~2y（CK 冷副本或 MinIO Parquet） | >2y 删除/脱敏后归档 |
| DWD/DIM | ≤ 1 年         | 1\~3 年                | 3\~5 年                       | 5y+ 归档       |
| DWS/ADS | ≤ 2 年         | 2\~5 年                | 5 年+                         | 长期保留         |
| 非结构化    | 按业务           | 按业务                   | MinIO LifeCycle 转冷           | 定义到档案管理规范    |

## 8. 数据口径与标准

- **一数一源**：指标、维度、主数据必须有唯一权威来源；
- **口径文档**：每个指标/维度必须在"指标中心"或"数据标准"中登记口径、公式、维度、频度、责任人、有效期；
- **变更留痕**：口径变更走审批 + 版本号；
- **示例**：在校生人数 = 学籍表中 `status = '在读'` 且 `register_flag = 'Y'` 的学生记录数，按 `dim_date` 口径日粒度统计。
