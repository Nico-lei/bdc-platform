# 02 数据库分库与表结构规范

## 1. 分库方案总览

按"业务域收敛 + 数据分层 + 强合规独立"三个维度分库，一期不按"每个微服务一个数据库"做物理拆分，统一口径如下：

### 1.0 一期物理库总览（与 `04-数据库详细设计/01-核心表结构清单.md`、`09-部署运维/01-离线部署方案.md` 保持一致）

| 物理库 | 引擎 | 用途 | 隔离方式 |
| --- | --- | --- | --- |
| `bdc_app` | PostgreSQL 15（业务实例） | 16 个业务服务的元数据 / 配置 / 任务库 | Schema 隔离，每个服务独占一个 Schema |
| `bdc_flowable` | PostgreSQL 15（业务实例） | Flowable 官方运行表 + `t_biz_flow_map` | 独立库（Flowable 自建） |
| `bdc_audit` | PostgreSQL 15（业务实例） | 审计日志主存储（只追加） | 独立库 + 触发器拒写改 |
| `bdc_dw` | PostgreSQL 15（**数仓实例，与业务实例物理分离**） | ODS / DWD / DIM 等结构化分层 | 按 Schema：`ods_*` / `dwd_*` / `dim` |
| `bdc_dw_olap` | ClickHouse 24.3 集群 | DWS / ADS 分析层 | CK 库 + 表前缀 `dws_*` / `ads_*` |

> 物理库一共 **4 个 PG 库 + 1 个 CK 库**。前端文档若出现 `bdc_uac`、`bdc_visual` 等写法，均表示 `bdc_app` 库下的 **逻辑模块 Schema**（即 `bdc_app.uac`、`bdc_app.visual`），不是独立物理库。

| Schema | 用途 | 所属服务 |

### 1.2 独立库（合规 / 引擎要求）

| 库 | 引擎 | 用途 |
| --- | --- | --- |
| `bdc_flowable` | PG | Flowable 7 BPMN 引擎运行库 + `t_biz_flow_map` |
| `bdc_audit` | PG | 审计日志主存储；按月分区；只允许 INSERT |

### 1.3 数据仓库库（数据层，独立实例）

| 库 / Schema | 引擎 | 用途 |
| --- | --- | --- |
| PG 库 `bdc_dw`，含 Schema `ods_*`, `dwd_*`, `dim` | PG 15 | 原始 + 明细 + 维度 |
| CK 库 `bdc_dw_olap`，含 `dws_*`, `ads_*` | CK 24.3 | 分析 + 应用 |

### 1.4 库级隔离策略

- 一期业务服务默认落在 `bdc_app`，按 Schema 隔离，**禁止跨 Schema 直连对方业务表**，跨服务必须走 OpenFeign 或 Kafka 事件；
- 数据仓库 `bdc_dw` 与业务实例 `bdc_app` **物理分离**（独立 PG 实例），避免数仓大查询影响业务；
- CK 仅存 DWS/ADS，不承担 OLTP；
- Flowable 与审计保留独立库，体现"引擎自治"与"审计三权分立"要求；
- 后续单 Schema 体量过大或合规升级时，可平滑剥离为独立库，不影响应用代码（仅改 DataSource 配置 + 跨库引用走 API）。

## 2. PG 分区策略

| 表类型 | 分区规则 | 示例 |
| --- | --- | --- |
| ODS 贴源表 | 默认不分区；仅超大表按 `_etl_ts` 月 RANGE 分区 | `ods_jwxt.t_student` / `ods_jwxt_t_log_2026m04` |
| DWD 事实表 | 默认不分区；当单表超过 5000 万行或 50GB 后按业务日期月 RANGE 分区 | `dwd_teach_course_select_d_2026m04` |
| DIM 拉链表 | 不分区（几万~百万级行即可） | `dim_person` |
| 审计日志 | 按 `create_time` 月 RANGE | `t_audit_log_2026m04` |
| 小表 / 参数表 | 不分区 | `bdc_app.config.*` |

设计原则：
- 优先减少分区数量和父子表层级，避免元数据、DDL、备份和统计信息维护复杂化；
- 一期禁止按天创建物理分区表，统一以“不分区”或“月分区”两档处理；
- 只有到达明显体量阈值的 ODS/DWD 大表才启用 `pg_partman` 自动维护月分区；其他表保持普通表。

## 3. ClickHouse 表引擎选择

| 场景 | 引擎 |
| --- | --- |
| 默认事实宽表 | ReplicatedMergeTree（2 副本） |
| 需要去重（按业务主键） | ReplicatedReplacingMergeTree |
| 预聚合 | ReplicatedAggregatingMergeTree |
| 明细流水（追加） | ReplicatedMergeTree |
| 本地临时 | MergeTree |
| 分布式查询 | Distributed 表 |

- 分区键：通常 `toYYYYMM(event_date)`；
- 排序键：查询最常用的维度列前置；
- TTL：自动过期/移冷；冷后移至 `storage_policy=cold_minio`。

## 4. 表结构设计规范

### 4.1 通用规则

1. 所有表必须有 **主键**（推荐 BIGINT 自增，或 UUID）；
2. 所有表必须有：`create_time`、`update_time`、`create_by`、`update_by`、`is_deleted`（软删）、`version`（乐观锁）、`tenant_id`（预留，默认 0）；
3. 所有字段必须 **NOT NULL DEFAULT**，除非业务允许 NULL（并在文档注明）；
4. 时间统一 `TIMESTAMPTZ`，禁止用 `TIMESTAMP`；
5. 金额统一 `NUMERIC(18,4)`；
6. 状态字段统一 `SMALLINT`（0/1/2…）+ 字典表；
7. 枚举字段在字典表中登记，严禁写死 SQL；
8. 字段注释必填（`COMMENT ON COLUMN`）；
9. 大字段（>1KB）单独拆子表，查询不带大字段；
10. JSON 字段选用 `JSONB` + GIN 索引。

### 4.2 命名规范

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 库 | `bdc_app` / `bdc_flowable` / `bdc_audit` / `bdc_dw` | `bdc_app` |
| Schema | 小写+下划线；业务模块按服务命名 | `uac`, `metadata`, `ods_jwxt` |
| 表 | `t_<domain>_<entity>` 业务表；`ods_/dwd_/dim_/dws_/ads_*` 数仓 | `t_user`, `dwd_stu_enroll` |
| 字段 | 小写+下划线 | `create_time`, `dept_id` |
| 索引 | `idx_<table>_<col1>_<col2>` | `idx_t_user_dept_id` |
| 唯一索引 | `uk_<table>_<col>` | `uk_t_user_login_name` |
| 外键 | `fk_<table>_<ref_table>` | `fk_t_user_t_dept` |
| 主键 | `pk_<table>` | `pk_t_user` |
| 序列 | `seq_<table>` | `seq_t_user` |

### 4.3 数据类型选择（PG）

| 场景 | 类型 |
| --- | --- |
| 主键 | `BIGINT`（推荐） / `UUID`（分布式场景） |
| 编号/工号 | `VARCHAR(32)` |
| 姓名 | `VARCHAR(64)` |
| 身份证 | `VARCHAR(64)` 密文（SM4） + `VARCHAR(32)` Hash（SM3 前 16B） |
| 手机号 | `VARCHAR(64)` 密文 + `VARCHAR(32)` Hash |
| 邮箱 | `VARCHAR(128)` 密文 + `VARCHAR(32)` Hash |
| 金额 | `NUMERIC(18,4)` |
| 布尔 | `BOOLEAN` |
| 长文本 | `TEXT` |
| 字典 | `SMALLINT` 或 `VARCHAR(32)` + 字典表 |
| JSON | `JSONB` |
| 地理 | `GEOMETRY`（PostGIS 扩展，按需） |

### 4.4 索引规范

1. 主键 = 聚簇索引；
2. 高频 WHERE/JOIN 字段必建索引；
3. 多字段组合索引遵循"最左前缀"，区分度高的在前；
4. JSONB 字段建 GIN；
5. 模糊查询 `LIKE '%xxx%'` 的前缀场景考虑 `pg_trgm` GIN；
6. 日志/审计类大表只建必要索引，避免写入放大；
7. **禁止**在密文字段上建索引（必须建在 Hash 上）。

### 4.5 外键策略

- 业务服务库内**可**使用外键约束；
- 跨服务数据关联**不**使用外键，仅作业务约束；
- 数据仓库 **不**使用外键（影响批量写入）。

### 4.6 分库分表引导

- 单表预估 **> 1 亿行 或 > 100 GB** 时，优先考虑：
  1. 按时间分区（PG Range 分区）；
  2. 按租户/学校分库（本期单校不触发）；
  3. 如仍不够，考虑迁移至 ClickHouse 或 ShardingSphere。
- 单表尚未达到上述规模前，不应提前引入日分区、二级分区或复杂分区模板。

## 7. ClickHouse 设计规范

- 分区数：每表不超过 1 万个分区（通常月分区已足够）；
- 单表列数：宽表列数 ≤ 300；
- 字符串用 `LowCardinality(String)` 代替枚举；
- 时间字段用 `DateTime` / `DateTime64`，不要存为 String；
- Join 以小表 + `join_use_nulls=1`，大表 Join 优先用 `Dictionary`；
- Distributed 表严格按"本地 + 分布式"双层；
- 每表必须配置 TTL，防止无限增长；
- 数据写入统一走批量（≥ 1 万行/批），禁止行级写入。

## 8. Flyway 数据库版本管理

- 每服务独立 `sql/` 目录；
- 版本号：`V<major>.<minor>.<patch>__<描述>.sql`；
- 不可修改历史版本，只能新增；
- 基础初始化：`V1.0.0__baseline.sql`；
- 字典数据 / 种子数据：`R__seed_*.sql`（可重复）；
- 索引/视图/存储过程：放在可重复脚本；
- DDL 与 DML 严格分离；
- 上线前由 DBA 审核并在预发环境验证。

## 9. 元数据管理

所有 DDL 上线后自动同步到 `bdc_metadata` 的资产目录：

```sql
INSERT INTO bdc_metadata.t_asset_table (...)
SELECT ... FROM pg_tables WHERE schemaname NOT IN ('pg_catalog','information_schema');
```

并定时扫描：
- 每日 02:00 扫描 PG 所有库 schema；
- 每日 02:30 扫描 CK ；
- 生成差异报告，变更告警到数据管理员。
