# 04 审批流程设计（BPMN + 动态路由）

## 1. 总体思路

- 引擎：**Flowable 7**（BPMN 2.0）；
- 采用"**流程模板 + 动态路由**"：同一模板按数据分级自动决定审批层级；
- 所有审批节点 **强制留痕**（进 bdc_audit），审计管理员可查。

## 2. 涉及流程清单（一期）

| 流程 Key | 名称 | 使用方 |
| --- | --- | --- |
| `market-apply` | 数据集市资源申请 | 全体师生 |
| `resource-publish` | 资源上架审批 | 数据 Owner |
| `api-publish` | API 上架审批 | API Owner |
| `indicator-change` | 指标口径变更 | 指标管理员 |
| `standard-change` | 数据标准变更 | 数据管理员 |
| `push-job-publish` | 推送任务发布 | 推送任务创建者 |
| `collect-job-sensitive` | 采集含敏感字段 | 采集任务创建者 |
| `kms-rotate` | KMS 密钥轮换 | 安全管理员 |
| `audit-export` | 审计导出 | 审计管理员 |
| `emergency-access` | 紧急访问 | 指定角色 |
| `col-class-confirm` | 字段分级复核 | 数据管理员 |

## 3. 动态路由核心：`market-apply`

### 3.1 输入参数

- `applicant`：申请人
- `resources`：资源清单及字段
- `batch_max_level`：批次内最高级别（L1~L4）
- `cross_dept`：是否跨部门
- `period_days`：授权时长
- `source_ip`

### 3.2 BPMN 流程

```
[开始] → [规则服务打分/定级]
       → 网关: level = ?
            ├─ L1 → [自动通过 + 审计] → [结束]
            ├─ L2 → [任务-部门数据管理员] → [结束]
            ├─ L3 → [任务-部门数据管理员] → [任务-数据管理员] → [结束]
            └─ L4 → [任务-部门数据管理员] → [任务-数据管理员] → [任务-安全管理员+二人复核] → [结束]
       → 跨部门 → 任何级别升 1 级
       → 超时 → 催办 + 通知上级
       → 拒绝 → [标记拒绝 + 通知申请人] → [结束]
```

### 3.3 审批矩阵（含时效）

| 级别 | 层级 | 审批人（角色） | 时效（工作日） | 超时 |
| --- | --- | --- | --- | --- |
| L1 | 0 | 自动 | 即时 | — |
| L2 | 1 | 资源所在部门的"部门数据管理员" | 1 | 催办上级 |
| L3 | 2 | ① 部门数据管理员 ② 数据管理员 | 2 | 每日催办 |
| L4 | 3 | ① 部门数据管理员 ② 数据管理员 ③ 安全管理员 + 二人复核 | 3 | 升级至安全领导小组 |

> **二人复核**：Flowable 使用 `MultiInstance` + `completionCondition=${approvedCount>=2}` 实现。

### 3.4 跨部门升一级的触发

- `resource.owner_org_id != applicant.org_id` 且非上级/下级关系 → `level += 1`（上限 L4）。

### 3.5 拒绝 / 撤销

- 任意节点拒绝流程结束；
- 申请人在未完成时可撤销；
- 同一申请 30 天内再提交自动提示"请勿重复申请"。

## 4. 通用配置（BPMN 变量）

| 变量 | 含义 |
| --- | --- |
| `maxLevel` | 批次最高级 |
| `crossDept` | 是否跨部门 |
| `approvers_l1`…`approvers_l3` | 每级候选人（动态计算） |
| `timeoutHours` | 超时小时 |
| `mfaRequired` | 是否需要 MFA |
| `doubleReview` | 是否双人复核 |

## 5. 候选人计算规则

- L1 部门数据管理员：资源 `owner_org_id` 对应的 `dept_dm` 角色用户（可多人）；
- L2 数据管理员：平台全局 `dataadmin` 角色；
- L3 安全管理员：平台全局 `secadmin` 角色；
- 未配置时退回到系统管理员（须预先避免）。

## 6. 审批端体验

### 6.1 待办列表
- 字段：流程名、申请人、申请主题、级别、到期时间、操作；
- 高级别用红标 + 倒计时；
- 移动端（微信/钉钉通知 + WAP 审批）。

### 6.2 审批详情
- 摘要：申请范围、字段、用途；
- 风险提示：含 L4 机密字段、批量过大、跨部门、异常 IP；
- 操作：通过 / 拒绝 / 转办 / 加签 / 评论；
- 必填理由：拒绝时 ≥ 20 字；

### 6.3 二人复核
- 复核人由 `secadmin` 选择另一名 `secadmin`；
- 双人均签方可进入下一步；
- 任一拒绝则驳回。

## 7. Flowable 集成要点

- 数据库：`bdc_flowable`；
- 启用 `flowable.history-level=full`；
- UI：`bdc-approval` 服务 + 前端 `flowable-bpmn-js` 定制；
- 集成：所有业务服务通过 `/api/approval/v1/process/start` 启动流程 + 传参；
- 事件：Flowable 发出 `BUSINESS_EVENT` → 消费后写审计 + 通知。

## 8. BPMN 示例（market-apply）

```xml
<bpmn:process id="market-apply" name="数据集市资源申请" isExecutable="true">
  <bpmn:startEvent id="start"/>
  <bpmn:serviceTask id="calcLevel" flowable:class="...CalcLevelDelegate"/>
  <bpmn:exclusiveGateway id="levelGate"/>
  <bpmn:sequenceFlow id="f1" sourceRef="levelGate" targetRef="endAuto">
    <bpmn:conditionExpression>${maxLevel == 1 and !crossDept}</bpmn:conditionExpression>
  </bpmn:sequenceFlow>
  <bpmn:sequenceFlow id="f2" sourceRef="levelGate" targetRef="l1UserTask">
    <bpmn:conditionExpression>${maxLevel &gt;= 2}</bpmn:conditionExpression>
  </bpmn:sequenceFlow>
  <bpmn:userTask id="l1UserTask" flowable:candidateUsers="${approversL1}" flowable:dueDate="PT24H"/>
  <bpmn:userTask id="l2UserTask" flowable:candidateUsers="${approversL2}" flowable:dueDate="PT48H"/>
  <bpmn:userTask id="l3UserTask" flowable:candidateUsers="${approversL3}" flowable:dueDate="PT72H">
    <bpmn:multiInstanceLoopCharacteristics isSequential="false"
       flowable:collection="${l3MultiList}" flowable:elementVariable="assignee">
      <bpmn:completionCondition>${approvedCount &gt;= 2}</bpmn:completionCondition>
    </bpmn:multiInstanceLoopCharacteristics>
  </bpmn:userTask>
  <!-- 条件连线 + 拒绝分支省略 -->
  <bpmn:endEvent id="endAuto"/>
  <bpmn:endEvent id="endApproved"/>
  <bpmn:endEvent id="endRejected"/>
</bpmn:process>
```

## 9. 通知策略

- 启动通知：@申请人；@下一级候选；
- 超时通知：每日一次；超过 3 次上报；
- 完成通知：@申请人 结果 + 授权生效时间；
- 审批全程留痕，通知记录进入消息中心。

## 10. 审批数据与授权联动

- 流程完成后 `ProcessCompletedListener` 写入 `bdc_uac.t_data_grant`；
- 生成数据访问票据（短期令牌或永久 scope）；
- 回调集市服务标记申请 `status=2 通过`；
- 回调文件服务预授权下载。

## 11. 其他流程概述（简写）

### 11.1 `resource-publish`
- 部门数据管理员 → 数据管理员 → 安全管理员（L3+）；
- 通过后资源 `status` 改为已上架。

### 11.2 `api-publish`
- 数据管理员 → 安全管理员；
- L1~L2 可免安全审批。

### 11.3 `indicator-change`
- 指标 Owner → 数据管理员；
- 发布后版本号自动升级。

### 11.4 `push-job-publish` / `collect-job-sensitive`
- 涉敏感字段：数据 → 安全；
- 涉出站（推送）：数据 → 安全 → 出口审批员（可选角色）。

### 11.5 `kms-rotate` / `audit-export`
- 安全管理员 + 审计管理员 双签；
- MFA 强制。

### 11.6 `emergency-access`
- 安全管理员 + 审计管理员 双签；
- 时效 2h，过后自动回收。

## 12. 审计要点

- 流程启动、节点流转、超时、催办、结束均写审计；
- 关键字段：`processKey, instanceId, taskId, assignee, decision, reason, ts, trace`；
- 审批导出需审计管理员授权。

## 13. 测试用例（节选）

| 场景 | 期望 |
| --- | --- |
| L1 公开，非跨部门 | 自动通过 |
| L2 本部门 | 1 级审批 |
| L3 本部门 | 2 级审批 |
| L3 跨部门 | 升级为 3 级 |
| L4 机密 | 3 级 + 双人复核 |
| 超时 | 自动催办 + 升级 |
| 拒绝 | 流程结束，申请人可查看原因 |
| 撤销 | 申请人撤销，通知审批人 |
| 二人复核 1 拒绝 | 流程驳回 |
