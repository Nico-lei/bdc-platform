# 02 权限模型（RBAC + ABAC + 数据权限）

## 1. 目标

- 覆盖 **菜单 / 接口 / 数据 / 操作** 四类权限；
- 支持库 / 表 / 行 / 列 四级数据权限；
- 满足动态场景（时段、IP、MFA、敏感级别）；
- 高性能：鉴权 P99 < 10 ms。

## 2. 模型关系

```
Org ─┬─ 包含 ──▶ User
     │            │
     │            └── 拥有 ──▶ Role（可多组织scope）
     │                            │
     │                            ├── 关联 ──▶ Menu / API（菜单+操作）
     │                            ├── 关联 ──▶ DataScope（资源访问级别）
     │                            ├── 关联 ──▶ RowRule（行过滤）
     │                            └── 关联 ──▶ ColRule（列可见/脱敏）
     │
     └── 资源属主 ──▶ Resource (Table/API/Dataset/File/Indicator/Tag/...)
```

## 3. 权限种类

### 3.1 菜单权限

- 控制导航菜单与按钮；
- 字段：`t_role_menu`；
- 前端登录后拉取菜单树 + perm 列表，进行渲染与按钮鉴权。

### 3.2 接口权限

- 控制哪些 URL 可访问；
- 由 `menu.perm` 或单独定义的 `api_perm`；
- Gateway 鉴权过滤器 + 服务端 `@PreAuthorize`。

### 3.3 数据权限（核心）

#### 3.3.1 访问级别

| Level | 名称 | 能力 |
| --- | --- | --- |
| 1 | 可见 | 仅在资产目录看到名称/描述 |
| 2 | 查询 | 可在线查询（受行列规则） |
| 3 | 下载 | 可下载（受行列规则 + 水印） |
| 4 | 修改 | 可更新数据（写库） |
| 5 | 管理 | 可授权他人 / 上下架 |

#### 3.3.2 行规则
- `where_template`：支持占位符 `:user.userId`, `:user.orgId`, `:user.orgPath`, `:user.deptIds`, `:term.current`, `:biz_date`；
- 运行时：在 SQL 解析阶段注入 where 条件；
- 多规则：AND 合并；
- 示例："仅允许访问本部门学生"：`student.org_id = :user.orgId`。

#### 3.3.3 列规则
- `visibility`：明文 / 脱敏 / 密文 / 不可见；
- `mask_rule_code`：引用 `bdc_security.t_mask_rule`；
- 前端渲染 + 后端双保险：
  - SQL 输出在 DAL 层替换；
  - API 返回 DTO 层二次脱敏；

#### 3.3.4 表/库级
- 对整张表、整个 Schema 直接授权/拒绝；
- 与列规则叠加：列规则在"允许查询"前提下生效。

### 3.4 操作权限

- 常见操作：`create`, `update`, `delete`, `publish`, `offline`, `grant`, `revoke`, `export`, `run`, `debug`...
- 与菜单按钮绑定；
- 高危操作（grant、KMS、审计导出）双因素 + 二人复核。

## 4. 身份到权限的计算

### 4.1 每次请求流程

```
Token 校验 → 取 user → 取 active roles → 合并：
  menus, perms, data-scope, row-rules, col-rules, env-rules
→ 缓存到 request context（Redis 缓存 5 min）
→ 各层消费
```

### 4.2 缓存策略

- Key: `perm:user:{userId}:v{version}`；
- Version：用户 / 角色 / 规则变更时 +1；
- TTL: 30 min；
- 变更事件：`event.uac.change` → 精准失效。

## 5. 动态属性（ABAC）

策略表达式（类 XACML，简化版）：

```yaml
policies:
  - id: admin-only
    target: {method: DELETE, path: /api/**}
    condition: user.roles contains "SYS_ADMIN" and env.ip in corp_ip_pool
    effect: allow
  - id: l4-download
    target: {action: download, level: L4}
    condition: user.mfa == true and env.hour between 8..18 and user.orgId == resource.ownerOrgId
    effect: allow
  - id: default-deny
    effect: deny
```

规则热更新：Nacos 配置发布 → 各服务监听刷新。

## 6. 权限数据结构（再看）

- `t_role`：角色
- `t_user_role`：用户角色（含组织 scope）
- `t_role_menu`：菜单授权
- `t_data_scope`：数据访问级别授权
- `t_row_rule`：行规则
- `t_col_rule`：列规则
- `t_data_grant`：来自集市申请的数据授权（=视为特殊 scope+row+col 组合）

详见 `04-数据库详细设计/01-核心表结构清单.md#A.2`。

## 7. 鉴权 SDK（bdc-common-security）

```java
@PreAuthorize("@perm.hasMenu('DATA_MARKET_RESOURCE') and @perm.canAccessLevel(#level)")
public void publishResource(Long id, int level) { ... }

@RowRule(table = "dwd_stu_enroll")  // 自动注入 where 条件
@ColRule(table = "dwd_stu_enroll")  // 自动脱敏列
public PageVO<Enroll> queryEnroll(QueryReq req) { ... }

@Audit(operation = "grant", objectType = "resource")
public void grant(Long resId, Long userId) { ... }
```

实现：MyBatis 插件（SQL 解析 + 重写）+ Spring AOP（DTO 脱敏）+ AuditAspect（事件埋点）。

## 8. 多组织与角色 Scope

- 用户可属多组织；
- 角色可绑定"组织 scope"（例：`审核员@信息办` 仅在信息办生效）；
- 鉴权时取"用户所属组织 ∩ 角色 scope"做并集计算。

## 9. 特殊场景

### 9.1 数据共享与临时授权
- 集市申请通过 → 写 `t_data_grant` → 生成等同 `t_data_scope + t_row_rule + t_col_rule` 的临时规则；
- 到期自动回收。

### 9.2 代表执行（Delegation）
- 允许管理员指派/代执行；
- 审计记录双人（操作者 + 委托人）。

### 9.3 紧急访问（Break-Glass）
- 应急账号，默认停用；
- 激活流程：安全 + 审计 双签；
- 使用期限 2 小时，自动回收并强制审计。

## 10. 鉴权性能

| 指标 | 目标 |
| --- | --- |
| 鉴权调用 P99 | < 10 ms |
| 规则数量 | 百万级 |
| 用户并发 | 1000 同时活跃 |

优化：
- 角色/规则索引；
- Redis 缓存；
- SQL 重写基于 AST 缓存；
- 鉴权结果缓存到当前请求上下文。
