# 03 微服务拆分与服务清单

## 1. 拆分原则

1. **领域驱动设计（DDD）**：以限界上下文（Bounded Context）为单位；
2. **单一职责**：服务不跨领域、不提供"什么都能干"的大杂烩；
3. **粒度适中**：初期避免过度拆分（一期控制在 15\~20 个服务），二期再按压力拆；
4. **无状态优先**：所有业务服务无状态，状态沉淀到存储组件；
5. **数据库按域隔离而非按服务分库**：一期采用少量共享业务库 + Schema 隔离，服务间仍禁止跨 Schema 直连对方业务表，必须通过 API/事件交互；
6. **共享内核仅限 DTO/常量/工具**：打成 `bdc-common` 二方库。

## 3. 服务依赖矩阵（简）

```
gateway ─┬─▶ auth
         ├─▶ uac (鉴权查询)
         └─▶ [所有业务服务]

业务服务 ─▶ uac (权限/组织)
        ─▶ metadata (字段元数据)
        ─▶ notify (消息)
        ─▶ audit (异步事件)
        ─▶ security (脱敏/加密/分级)

collect ─▶ metadata, warehouse
govern  ─▶ collect, warehouse, metadata
push    ─▶ warehouse, metadata
service-api ─▶ warehouse, metadata
market  ─▶ warehouse, metadata, approval, file
visual  ─▶ service-api, metadata
indicator ─▶ metadata, warehouse
tag ─▶ metadata, warehouse
mdm ─▶ metadata, warehouse
approval ─▶ uac, notify
```

## 4. 通信方式

| 场景          | 通信                                   |
| ----------- | ------------------------------------ |
| 前端 ↔ 网关     | HTTPS + JSON                         |
| 网关 ↔ 业务服务   | HTTP + OpenFeign                     |
| 业务 ↔ 业务（同步） | OpenFeign + Sentinel                 |
| 业务 ↔ 业务（异步） | Kafka 事件 + `bdc-common-event` 统一事件结构 |
| 业务 ↔ DS 调度  | DS REST API                          |
| 业务 ↔ Flink  | REST + Flink Application Mode 提交包    |
| 分布式事务       | Seata AT（强一致短事务） / SAGA（长流程）         |

## 6. 代码仓库规划

单仓 Monorepo（`bdc-platform`）+ Gradle Composite，子模块结构：

```
bdc-platform/
├── bdc-bom/                     # BOM 版本统一
├── bdc-common/                  # 二方库
│   ├── core/  security/  audit/  feign/  event/  web/  db/  cache/
├── bdc-gateway/
├── bdc-services/
│   ├── auth/  uac/  metadata/  mdm/  indicator/  tag/
│   ├── collect/  warehouse/  govern/  push/  service-api/
│   ├── market/  approval/  visual/  security/  audit/  notify/
│   ├── scheduler-adapter/  file/
├── bdc-flink-jobs/              # Flink 作业（CDC、实时 ETL、质量实时检）
├── bdc-datax-plugins/           # 定制 DataX Reader/Writer（达梦/金仓增强、国密写）
├── bdc-frontend/                # Vue 前端 Monorepo
│   ├── portal/  admin/  screen/  market/  designer/
├── bdc-deploy/                  # Docker Compose / 初始化 SQL / 脚本
└── bdc-docs/                    # 项目内 docs（与本设计保持同步）
```

## 7. 服务交付清单（每个微服务需提供）

1. Java 源码 + Gradle 构建脚本
2. Dockerfile（多阶段构建；base 镜像为私有 `bdc/base-jdk17`）
3. Docker Compose 部署文件 + `.env.prod` / `.env.test`
4. 初始化 SQL：`sql/V1.0.0__init.sql` 到 `sql/Vx.y.z__*.sql`（Flyway）
5. OpenAPI 3 规范文件 `api/service-xxx.yaml`
6. 单元测试覆盖 ≥ 60%，集成测试覆盖核心主流程
7. README：服务职责、关键表、依赖服务、常见问题
8. 部署清单：CPU/内存/磁盘/副本数

## 8. 非功能属性目标（每服务）

| 指标                | 建议值                      |
| ----------------- | ------------------------ |
| 启动时间              | ≤ 30 s（冷启动）              |
| JVM Heap          | 默认 2G（根据服务调整）            |
| P99               | ≤ 200 ms（读）/ ≤ 500 ms（写） |
| 副本数               | 2+（核心 3+）                |
| 健康检查              | /actuator/health         |
| Graceful Shutdown | 默认启用 30s                 |
