## 2. 架构风格与原则

| 原则    | 说明                                |
| ----- | --------------------------------- |
| 分层解耦  | 每一层只依赖下一层的接口，跨层禁止直连               |
| 前后端分离 | 前端 Vue SPA，后端 REST + 国密签名         |
| 微服务化  | 以领域（DDD）为边界，单一职责、独立部署             |
| 事件驱动  | 跨域异步用 Kafka 事件，强一致场景才用 Seata      |
| 数据驱动  | 所有配置（采集、调度、血缘、权限）元数据化，界面化         |
| 国产化优先 | 算法国密、OS 国产、数据库 PG、中间件全 Apache 2.0 |
| 安全内嵌  | 安全从网关→服务→存储→备份全链路贯穿（Zero-Trust）   |
| 可观测性  | 日志/指标/链路三件套 + 业务埋点                |

## 3. 关键运行时流程

### 3.1 数据全链路流程

```
业务系统 ──(Flink CDC/DataX)──▶ Kafka ──▶ Flink ETL ──▶ PostgreSQL(ODS)
                                                              │
                      定时(DS) ──▶ SQL/Flink 清洗 ──▶ PostgreSQL(DWD/DIM)
                                                              │
                      定时(DS) ──▶ 聚合 ──▶ ClickHouse(DWS/ADS)
                                                              │
                                           ┌──────────┬───────┴───────┐
                                           ▼          ▼               ▼
                                      数据服务API  数据大屏        数据集市
                                           │
                                           ▼
                                      外部系统 / 数据推送
```

### 3.2 SSO 登录流程

```
用户 ──▶ 门户(未登录) ──302──▶ CAS Server ──(Ticket)──▶ 门户回调
  │                                                         │
  │                                                         ▼
  └────── Cookie JSESSIONID/Token ──▶ 网关 ──(校验)──▶ 后端服务
```

### 3.3 数据申请-审批-使用流程

```
集市浏览 → 加购物车 → 提交申请(Flowable) → 部门→数据→安全 审批
        → 授权(写入权限服务) → 用户查看/下载(水印)
        → 审计 & 定期回收
```

## 5. 逻辑架构视图（领域分布）

按 DDD 拆 **9 个限界上下文**，物理层落到 **20 个微服务**（详见 `03-微服务拆分与服务清单.md`）：

| # | 限界上下文 | 微服务 |
| --- | --- | --- |
| 1 | 采集域（Collect） | `bdc-collect` |
| 2 | 仓库域（Warehouse） | `bdc-warehouse`、`bdc-file`（MinIO 文件代理） |
| 3 | 治理域（Govern） | `bdc-govern`、`bdc-scheduler-adapter`（DS Facade） |
| 4 | 推送域（Push） | `bdc-push` |
| 5 | 服务域（Service） | `bdc-service-api` |
| 6 | 集市域（Market） | `bdc-market`、`bdc-approval`（Flowable 审批引擎） |
| 7 | 可视化域（Visual） | `bdc-visual` |
| 8 | 安全域（Security） | `bdc-auth`、`bdc-uac`、`bdc-security`、`bdc-audit`、`bdc-gateway`（接入侧鉴权 / 限流 / 签名）、`bdc-notify`（通知中心） |
| 9 | 元数据域（Metadata） | `bdc-metadata`（资产目录 + 数据标准）、`bdc-mdm`、`bdc-indicator`、`bdc-tag` |

> 注：9 个限界上下文 = 业务边界视角；20 个微服务 = 落地视角。`gateway` 与 `notify` 虽然是横切组件，按"功能就近原则"归入安全域；`scheduler-adapter`、`file` 同理归入治理域 / 仓库域。

## 6. 部署视图（简版）

详见 `04-部署架构与网络设计.md`。生产环境推荐 **Docker 容器 + Docker Compose** 分角色部署，单机房多安全域隔离，全离线 Harbor 私仓拉镜像。

## 7. 架构决策记录（ADR 摘要）

| ID      | 决策                   | 理由               | 备选             |
| ------- | -------------------- | ---------------- | -------------- |
| ADR-001 | OLAP 选 ClickHouse    | 大屏/聚合首选；运维简单     | Doris/GP       |
| ADR-002 | CDC 选 Flink CDC      | 全量+增量一体，生态最佳     | Debezium/Canal |
| ADR-003 | 批采集选 DataX           | 插件多，支持达梦/金仓      | SeaTunnel      |
| ADR-004 | 调度选 DolphinScheduler | 国产中文友好           | Airflow/XXL    |
| ADR-005 | 工作流选 Flowable        | BPMN 2，轻量，中文文档充足 | Camunda        |
| ADR-006 | 对象存储选 MinIO          | S3 协议+EC+国密友好    | Ceph           |
| ADR-007 | 密码全国密                | 等保三级过评           | AES/RSA        |
| ADR-008 | 网关选 Gateway          | 与 SCA 体系无缝       | Kong/APISIX    |
| ADR-009 | 服务通信选 OpenFeign      | 同步为主，简单易读        | gRPC           |
| ADR-010 | 事件总线选 Kafka          | 数据与业务事件统一        | RocketMQ       |

详细见 `06-安全与合规/` 与各模块详设。
